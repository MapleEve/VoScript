执行摘要

VoScript 不是在补“转录”空白，而是在补“身份感知的组织内语音后端”空白

VoScript 当前最合理的位置，不是基础模型平台、不是通用 runtime、不是泛化会议 SaaS，而是固定团队/组织内部“长期声纹库 + 真名级会后逐字稿 + 工作流集成”的自托管身份感知语音后端。

当前主轴：Identity-aware backend 1.0 前：继续收口，不平台化 1.0 后：meeting speech backend vision
关键判断：VoScript 填补的不是“没人做转录”的空挡；真正空挡只在固定人群高频录音整理场景中成立。若把它讲成所有转录用户都需要，就会滑向伪需求。
产品轴

先分清“它卖什么”，再比较“它会什么”

P1 基础模型 / 训练平台

卖模型、训练、微调、评测、模型生态。代表：FunASR。

P2 通用推理 runtime / SDK

卖本地推理、多任务 runtime、跨平台 SDK。代表：sherpa-onnx。

P3 流式增量转录引擎

卖低延迟 partial / revised / final 和在线回看修正。代表：Whisper-Streaming。

P4 实时 ASR 服务层

卖实时 server、WebSocket/RTC、API 兼容层。代表：WhisperLiveKit、Deepgram、AssemblyAI、Gladia。

P5 通用转录 API / 异步语音后端

卖文件上传、异步 job、查询、导出。代表：Google STT、Azure Speech、腾讯云语音，以及 VoScript 的一部分。

P6 身份感知语音后端

卖长期 voiceprint memory、真名级归因、跨会话识别。VoScript 当前主定位。

P7 会议/录音工作流后端

卖录音整理、会后回看、导出、工作流接入。VoScript 当前次定位 / 1.0 前主收口方向。

P8 会议 SaaS / 会议纪要产品

卖终端产品体验、协作、纪要、待办、组织工作台。代表：飞书妙记、腾讯会议AI、讯飞听见、通义听悟、Otter、Fireflies、Notta。

矩阵解读

VoScript 处在两类成熟产品之间的缝隙

左侧：基础平台

FunASR、sherpa-onnx、Riva、Azure、Google 这类对象强在能力面与底层分发，不以“长期实名 speaker + 会后整理工作流”作为主叙事。

中间：实时服务

Whisper-Streaming、WhisperLiveKit、Deepgram、AssemblyAI、Gladia 强在 streaming、partial/final、WebSocket 和实时会话，不主打组织内长期 speaker memory。

右侧：会议产品壳

飞书、腾讯会议、钉钉、Otter、Fireflies、Notta、讯飞听见、通义听悟更像终端产品与协作体验，通常不是自托管后端，也不以持久化声纹库为主卖点。

定位缝隙：VoScript 最有机会占住的是“模型能力层”与“完整会议 SaaS”之间的中间层：既比通用转录 API 更懂身份与工作流，又比会议 SaaS 更可自托管、更可嵌入、更像可复用后端。
定位坐标轴

二维坐标轴比一维产品轴更能看清 VoScript 身位

坐标定义

  • X 轴 = 从基础能力层 → 产品工作流层
  • Y 轴 = 从匿名转录/通用能力 → 身份感知/长期记忆

这个坐标不是比较“谁更强”，而是比较“谁站在哪一层卖什么”。因此 VoScript 不该再被误投到基础模型、通用 runtime 或泛会议 SaaS 身位。

一句话结论

VoScript 当前身位 = 靠右上，但仍低于完整会议产品层；更准确是“身份感知语音后端”。

post-1.0 的方向应该是继续向右、略向上：变成更强的工作流后端 / meeting speech backend，而不是向基础模型平台、通用 runtime 或泛会议 SaaS 漂移。

不该去的身位

  • 停在左中，变成通用 runtime / SDK

  • 右上：Otter / Fireflies / Notta
  • VoScript 当前：右上内侧的身份感知语音后端
当前定位

VoScript 当前最准确的表达

VoScript = self-hosted、identity-aware 的 meeting / recording transcription backend。
  • self-hosted：强调部署边界、数据控制与组织级可落地性
  • identity-aware：强调不只是识别内容，更重视“是谁说的”
  • meeting / recording：强调围绕录音整理场景，而不是覆盖所有语音任务
  • backend：强调它是工作流与产品的后端能力层，不是纯 SDK，也不是完整会议 SaaS

现在最强的主轴

  • Identity-aware speech backend
  • General transcription API backend
  • Meeting / recording workflow backend

明确不是当前主轴

  • 通用训练平台
  • 通用 runtime SDK
  • 成熟实时引擎
  • 完整会议协作 SaaS
空挡与伪需求

VoScript 填补的是“组合空挡”，不是单点空挡

它填补的空挡

  • 长期声纹库
  • 真名级会后逐字稿
  • 组织内持续复用
  • 异步 job + HTTP API
  • 可嵌入现有知识库/纪要/协作系统

伪需求风险

  • 把它讲成“所有转录用户都需要”
  • 把它讲成“更好的转录”而不是“更少的人名修正成本”
  • 忽略目标用户必须是固定团队/组织
  • 忽略维护 speaker 库本身有门槛
强需求前提：同一批人反复开会、反复录音、反复整理；如果没有这个前提，VoScript 的长期声纹库价值会迅速下降。
差异判断

VoScript 与 sherpa-onnx 的区别不在“功能多少”，而在抽象层不同

sherpa-onnx 更像什么

  • 通用语音 runtime / 多任务基础平台
  • 本地推理、跨设备、跨语言绑定
  • 关注 task / runtime / deployment target

VoScript 更像什么

  • 会议/录音整理场景的身份感知语音后端
  • 关注 transcription job / enrolled speaker / voiceprint library / named transcript
  • 关注长期身份、结果沉淀、工作流集成
一句话：sherpa-onnx 更偏“通用语音 runtime / 多任务基础平台”；VoScript 更偏“会议与录音场景的应用后端”。
1.0 前后边界

1.0 前先收口，1.0 后再扩大

1.0 前

  • 继续把 VoScript 收到稳定
  • 收口目录架构、contract、artifact、speaker 事实源
  • 收口 backend interface、scheduling / observability
  • 不把项目写成通用语音平台

1.0 后

  • 可插拔推理后端
  • batch / streaming 协议统一
  • speaker memory
  • transcript state machine
  • orchestration / product API
但即便是 post-1.0，也不应变成：另一个 FunASR、另一个 sherpa-onnx、另一个通用训练平台、另一个 model zoo、另一个全任务语音中台。
战略建议

应该继续做什么，不要变成什么

应该继续做的

  • 把 VoScript 收到 1.0
  • 强化 identity-aware 这条主轴
  • 把 HTTP API + async job + artifacts + speaker memory 做成稳定后端
  • 1.0 后再考虑 streaming / transcript state machine / meeting speech backend

不要变成什么

  • 另一个 FunASR
  • 另一个 sherpa-onnx
  • 泛化“什么转录都做”的平台
  • 为了追实时而丢掉 identity-aware 护城河
  • 为了看起来更大而把需求讲成泛需求
最终一句话：VoScript 最合理的长期路线，不是成为“最全的语音平台”，而是成为固定团队/组织内部，围绕会议与录音整理场景的、可自托管的身份感知语音后端。
