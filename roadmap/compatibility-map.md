# Compatibility Map

## 目标

在 1.0 前，VoScript 的首要任务不是不断引入新名词，而是把已经形成用户依赖的持久化与结果协议稳住。

本文件记录 1.0 前默认冻结的兼容边界。

## 冻结原则

### 1) `result.json` 冻结

- `result.json` 是转录完成态的核心持久化结果
- 在 1.0 前，字段只能谨慎增量扩展，不做随意重命名或语义漂移
- 已存在字段的含义必须保持稳定，尤其是 transcript、segments、speaker 相关字段
- 接入方一旦开始依赖 `result.json`，升级不应迫使其重写基础解析逻辑

### 2) `status.json` 冻结

- `status.json` 是 job lifecycle 的持久化状态入口
- 1.0 前保持“重启后可恢复查询”的基本语义不变
- 状态机可以补充更清晰的枚举或元信息，但不能破坏现有轮询心智
- 任何新状态都应保证旧客户端仍能以保守方式处理

### 3) `speaker_label` 冻结

- `speaker_label` 继续作为单次录音内部的原始说话人标签
- 它不是 display name，也不是全局 speaker id
- 自动匹配到历史声纹时，`speaker_name` 可以变化，但 `speaker_label` 的角色不能漂移
- 任何 enroll / relabel / transcript edit 语义都必须继续尊重这个边界

### 4) `voiceprints.db` 冻结

- `voiceprints.db` 继续作为 1.0 前的默认持久化声纹库
- 可以做 schema 演进与迁移，但不能把它从“稳定、可备份、可迁移的本地数据库”改成瞬态缓存或不可预期结构
- 升级路径必须优先考虑已有部署的数据安全

## 兼容性要求矩阵

| 对象 | 1.0 前默认原则 | 允许变化 | 不允许变化 |
| --- | --- | --- | --- |
| `result.json` | 读路径稳定、核心字段语义稳定 | 增加新字段、补充元信息 | 随意改名、删除核心字段、重定义既有字段 |
| `status.json` | 轮询入口稳定、重启后查询语义稳定 | 增加诊断字段、细化状态描述 | 改掉基本读取方式、破坏旧轮询逻辑 |
| `speaker_label` | 单录音内部原始标签稳定 | 文档补充、错误提示更明确 | 改成 display name / 全局 id / 可漂移别名 |
| `voiceprints.db` | 本地持久化声纹库定位稳定 | 可控 schema migration、性能优化 | 隐式丢数据、改成不透明不可迁移存储 |

## 面向版本的解释

- v0.7.2: 把冻结原则写清楚
- v0.8.0: 用架构边界减少未来破坏式调整的概率
- v0.9.0: 把 API 与状态机语义整理到更稳定
- v0.9.5: 做冻结预演
- v1.0.0: 将这些兼容原则视为正式产品承诺的一部分

## 备注

这里的“冻结”不是指永远不能升级，而是指：

- 先保证已有用户不会因为一次小版本升级就失去可预期行为
- 真要变，必须显式迁移、显式文档、显式说明破坏范围
- 在 1.0 前，兼容性优先级高于引入炫技式新能力
