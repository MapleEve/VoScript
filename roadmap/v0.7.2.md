# v0.7.2 Planning + Architecture Foundation

## 版本定位

v0.7.2 是一次收口型版本：一方面把路线、边界、兼容性与语言讲清楚；另一方面完成 pipeline/provider/infra 的基础架构铺垫，为后续 v0.8 的可配置 provider 与更完整插件化打底。

这不是能力面扩张版本。v0.7.2 不把 provider preset / API 级 provider 选择、streaming、speaker memory 讲成当前交付能力。

## 这一阶段要解决的问题

- roadmap 还没有形成体系化表达
- 1.0 前主线与 1.0 后 vision 容易被混写
- README 缺少简洁但明确的路线说明
- 外部容易把 VoScript 误解成将要扩成通用语音基础设施
- 与 sherpa-onnx 这类项目的区别没有被明确表达

## 本版本 planning 目标

### 1. 建立 roadmap 文档树
- 建立总览、version map、compatibility map、版本文档、post-1.0 vision 文档
- 让后续版本讨论有固定落点

### 2. 把 1.0 前主线收紧
- 保持当前主叙事：自托管 GPU 转录服务 + 持久化声纹 + HTTP API
- 明确 1.0 前先把产品与协议打稳
- 不把更大的愿景混入当前版本承诺

### 3. 写清长期方向但单独存放
- 单独新增 vision-post-1.0.md
- 明确统一语音应用后端 / meeting speech backend 的方向
- 同时明确不做另一个 FunASR，不做通用训练平台 / model zoo / 全任务语音中台

### 4. 写清与 sherpa-onnx 的区别
- 区分“通用 runtime / 多任务基础平台”与“会议/录音场景应用后端”
- 避免 scope 被误读

### 5. 完成架构基础铺垫
- 将 pipeline、provider、infra、application、voiceprints 的目录边界落到代码里
- 保持现有 HTTP API 与持久化结果兼容
- 为 v0.8 provider 可配置化和观测指标预留结构，但不提前公开为产品承诺

## 不在本版本承诺的内容

- 不承诺在 v0.7.2 直接实现 streaming
- 不承诺在 v0.7.2 引入新的通用语音任务矩阵
- 不承诺在 v0.7.2 改写现有产品定位
- 不承诺在 v0.7.2 暴露 provider preset 或 API 级 provider 选择
- 不承诺在 v0.7.2 完成 speaker memory 产品化

## 已知取舍

- 当前完整转录 pipeline 仍运行在既有 GPU 串行保护内。该策略优先降低架构重构后的并发风险，但会让部分 CPU/IO 阶段被一起串行化，吞吐可能偏保守。
- CPU / IO / GPU stage 的锁边界拆分、stage latency 指标和 GPU wait-time 指标延后到 v0.8。

## 交付物

- roadmap 文档体系建立完成
- README / README.en 完成最小但清晰的路线补充
- compatibility 冻结原则形成文档化表述
- pipeline / providers / infra / application / voiceprints 基础目录边界形成
- healthcheck、版本元信息、发布说明与实际候选行为对齐

## 完成标准

- 团队成员能清楚区分 1.0 前承诺与 1.0 后 vision
- 外部读者能从 README 快速找到 roadmap 与 vision
- 文档中明确 VoScript 与 sherpa-onnx 的长期方向差异
